# 02. 설계 원칙 및 패턴

CheckMo는 유지보수와 확장이 용이하도록 여러 소프트웨어 공학 원칙을 준수하여 설계되었습니다. 이 문서는 프로젝트 전반에 흐르는 핵심 철학과 구현 패턴을 설명합니다.

## 1. 핵심 설계 원칙 (Core Principles)

### SoC (Separation of Concerns, 관심사 분리)
프로젝트는 역할에 따라 레이어를 엄격히 분리합니다.
*   **Component**: 오직 UI 렌더링과 사용자 이벤트 전달에 집중합니다.
*   **Hook**: 비즈니스 로직, 데이터 페칭 상태, 이벤트 핸들러를 관리합니다.
*   **Service**: 외부 서버와의 통신(API 호출)만을 담당합니다.
*   **Store**: 여러 컴포넌트에서 공유해야 하는 최소한의 전역 상태를 관리합니다.

### SoP (Separation of Protocols, 프로토콜 분리)
외부 인터페이스(백엔드 API)와 내부 비즈니스 로직을 분리합니다.
*   `src/services/mappers` 디렉토리를 통해 서버 응답(DTO)을 프론트엔드에 최적화된 모델로 변환하는 구조를 지향합니다. (현재 고도화 단계로 도입 예정)

### SOLID (객체 지향 5대 원칙 적용)
*   **SRP (단일 책임 원칙)**: 각 서비스 파일과 훅은 하나의 도메인 또는 역할만 수행합니다. (예: `authService`는 인증만, `useMemberQueries`는 회원 정보 조회만)
*   **OCP (개방-폐쇄 원칙)**: `apiClient`와 같은 공통 모듈은 확장에 열려 있고, 기존 코드 수정 없이 옵션 추가가 가능한 구조입니다.

## 2. 데이터 관리 패턴 (Data Management)

### Server State (TanStack Query)
서버 데이터는 전역 상태가 아닌 '캐시된 데이터'로 취급합니다.
*   **Query Key Factory**: `useMemberQueries.ts`의 `memberKeys`와 같이 키 관리 객체를 두어 오타를 방지하고 일관성을 유지합니다.
*   **Custom Hooks**: 모든 API 호출은 커스텀 훅으로 래핑하여 컴포넌트에서 로딩/에러/데이터 상태를 간결하게 사용합니다.

### Client State (Zustand)
사용자 인증 상태, 모달 열림 여부 등 UI 중심의 전역 상태는 Zustand를 사용합니다.
*   **Minimalism**: 전역 상태는 꼭 필요한 경우에만 사용하며, 가능한 로컬 상태(`useState`)나 URL 상태를 우선시합니다.

## 3. 코드 패턴 예시 (Implementation Pattern)

### API 서비스 작성 규칙
```typescript
// src/services/memberService.ts
export const memberService = {
  getProfile: async () => {
    return await apiClient.get<ApiResponse<User>>(AUTH_ENDPOINTS.PROFILE);
  },
};
```

### 쿼리 훅 작성 규칙
```typescript
// src/hooks/queries/useMemberQueries.ts
export const useProfileQuery = (enabled: boolean = true) => {
  return useQuery({
    queryKey: memberKeys.profile(),
    queryFn: () => memberService.getProfile(),
    enabled,
  });
};
```

### 컴포넌트 사용 규칙
```tsx
// src/components/profile/ProfileDetail.tsx
const { data, isLoading } = useProfileQuery();
if (isLoading) return <Spinner />;
return <div>{data?.nickname}</div>;
```

## 4. KISS & DRY

*   **KISS (Keep It Simple, Stupid)**: 불필요한 추상화를 지양하고, 코드를 읽었을 때 의도가 즉시 파악되도록 작성합니다.
*   **DRY (Don't Repeat Yourself)**: 반복되는 UI 패턴은 `base-ui`로 공통화하고, 중복 로직은 커스텀 훅으로 추출합니다.
