# 06. 미결 사항 및 개선 제안

본 문서는 현재 코드의 인수인계 과정에서 도출된 한계점 및 기술 부채를 명시하고, 향후 프로젝트 고도화 시 우선적으로 처리해야 할 개선 제안을 담고 있습니다. 코드를 임의로 수정하지 않고(최소주의 원칙), 발전 방향을 제시하는 데 목적이 있습니다.

## 1. 미결 사항 및 잔재 코드 (Cleanup Required)

### `src/proxy.ts` 관련
*   **현황**: 해당 파일은 과거 로컬 네트워크 환경 개발 중 CORS 오류를 우회하기 위해 도입 시도했던 프록시 설정의 잔재입니다.
*   **문제점**: Next.js의 공식 미들웨어 표준 파일명인 `middleware.ts`와 다르며, 현재 시스템에서 참조되거나 올바르게 동작하지 않고 있습니다.
*   **처리 방안**: 프로젝트 구조의 명확성을 위해 연관된 코드가 없음을 확인한 후 완전히 삭제할 것을 권장합니다.

## 2. 아키텍처 개선 제안 (Architectural Improvements)

### API 응답 매핑 로직 (SoP 적용)
*   **현황**: `src/services/mappers` 디렉토리가 생성되어 있으나 로직이 비어있으며, 백엔드 API 응답(DTO)을 프론트엔드 UI에서 그대로 소비하고 있습니다.
*   **문제점**: 백엔드의 데이터 스펙(필드명, 구조 등)이 변경될 경우, 이를 사용하는 모든 UI 컴포넌트들을 일일이 찾아 수정해야 하는 파급 효과가 발생합니다. (결합도 높음)
*   **해결 방안 (고도화 단계)**: `mappers` 디렉토리를 활용하여 서버에서 넘어온 DTO를 프론트엔드 전용 모델(Model)로 변환(Parsing/Mapping)하는 계층을 구현해야 합니다. (프로토콜 분리 원칙)

### apiClient 내 옵션 일관성
*   **현황**: `src/lib/api/client.ts`의 `RequestOptions` 인터페이스에 `silentAuthError?: boolean` 속성이 정의되어 있습니다. (실제로 `authService.getProfile` 등에서 사용 중)
*   **문제점**: 하지만 정작 `request` 함수 내부 구현체에서는 해당 속성을 확인하여 401 에러 인터셉터를 건너뛰는(우회하는) 로직이 누락되어 있습니다. 즉, 선언만 되어있고 작동하지 않습니다.
*   **해결 방안**: 향후 인증 오류(401)가 발생해도 강제로 로그아웃 처리나 에러 토스트를 띄우지 말아야 하는 조용한 API 호출이 명확히 필요해진다면, 해당 로직을 `apiClient` 내부에 올바르게 구현해야 합니다. 불필요하다면 옵션 정의를 삭제하여 혼동을 방지해야 합니다.

## 3. 요약

현재 CheckMo 프로젝트는 Next.js App Router, TanStack Query, Zustand 등의 최신 기술과 SoC 원칙을 잘 버무려 훌륭한 유지보수성을 갖추고 있습니다. 향후 위의 개선 사항들(특히 데이터 매핑 분리)이 순차적으로 적용된다면 더욱 견고한 프론트엔드 아키텍처로 발전할 것입니다.
